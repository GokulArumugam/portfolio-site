# Payments Reconciliation Engine

## Problem

Processor settlements, internal ledger records, fees, and reversals rarely arrive in the same shape or at the same time. The system needs to identify what is matched, what needs review, and what can safely be closed.

The reconciliation model tracks each source record through staging, deterministic matching, exception handling, and close.

## Key Decisions

- Model matching as versioned rules, not one opaque query.
- Preserve links from every match back to its source evidence.
- Separate automatic closure from human-reviewed exceptions.

## Failure Modes

Expected failures include duplicate files, partial settlement delivery, changed processor layouts, and late reversals. Runs are idempotent, input snapshots are retained, and close is guarded by explicit completeness checks.

## Cost

The workload is intentionally batch-oriented. Object storage holds source artifacts, while warehouse spend is controlled through incremental partitions and a bounded review queue.
