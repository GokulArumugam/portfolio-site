# Payments Reconciliation Engine

A deterministic reconciliation engine for the two views of payment traffic that matter at daily close: what the switch says happened, and what the bank says settled.

## Problem

The switch ledger is real-time and authoritative for payment attempts. The bank settlement file arrives at T+1 and is authoritative for money movement. They should describe the same transactions, but real operations create drops, duplicate posts, fee or rounding differences, status disagreements, late settlements, and malformed references. The job is not to make the totals look plausible: it is to classify every disagreement with evidence that an operator can audit.

The generator produces the two ledger views plus a separate defect manifest. The engine never reads that manifest: five numbered DuckDB passes normalize inputs, exclude repeated keys, perform exact matching, attempt a deliberately constrained fuzzy rescue, then classify every residual. The manifest is used only after the run to verify what the engine found.

## Key decisions

1. **[ADR-0001: deterministic multi-pass matching](https://github.com/GokulArumugam/payments-reconciliation/blob/main/docs/adr/0001-deterministic-multipass-matching.md).** Ordered rule passes make every result explainable. If constrained fuzzy matching has more than one candidate, it emits `AMBIGUOUS`; it never guesses.
2. **[ADR-0002: DuckDB SQL as the core](https://github.com/GokulArumugam/payments-reconciliation/blob/main/docs/adr/0002-duckdb-sql-core.md).** The numbered SQL files are the matching logic, while orchestration only loads inputs and writes outputs. That makes the same rules runnable on a laptop, in CI, and in DuckDB-WASM.
3. **[ADR-0003: ground-truth oracle](https://github.com/GokulArumugam/payments-reconciliation/blob/main/docs/adr/0003-ground-truth-oracle.md).** The generator emits a defect manifest separately from the reconciler inputs. Verification compares discovered `(txn_id, defect_class)` pairs with that oracle instead of relying on a small hand-written fixture set.

## Correctness contract

The generator emits a defect manifest that acts as the oracle. It is deliberately isolated from `switch_ledger` and `bank_settlement`, then compared with the reconciler outputs after the run. The contract is strict: every injected defect must be rediscovered, and no clean transaction may be reported as a defect.

Verified on a laptop: **1M pairs reconciled in 4.7s**, with **100% recall across all 35,000 injected defects** in the seven defect classes and **zero false positives**. The 5M-scale measurement is **5M pairs in 217s on the same laptop (memory-bound alongside the streaming pipeline VM; 175,000 defects, still 100% recall, zero false positives)**.

## Failure modes

| Failure mode | Engine behavior |
| --- | --- |
| Dedup on both sides | A repeated UTR on either or both sides is classified as `DUPLICATE` and excluded from later matching passes, preventing double matching. |
| `AMBIGUOUS` fuzzy candidates | More than one constrained candidate becomes an `AMBIGUOUS` exception. The engine must never guess. |
| Whole-file-missing | An empty bank file causes all remaining switch transactions to classify as `MISSING_IN_BANK`; the batch summary can raise a file-level completeness alarm. |
