# Real-time Wikipedia Edits Pipeline

## Problem

Wikimedia publishes a continuous stream of edit events. The system needs to ingest that stream, preserve the raw record for replay, and make a well-shaped version available to downstream analysts with bounded freshness.

Raw events land first, then a normalization step produces query-friendly records while retaining an immutable replay source.

## Key Decisions

- Treat the source payload as an append-only contract, not a database schema.
- Keep event time and processing time separate for late-arrival analysis.
- Make idempotency explicit at every boundary so replays are routine rather than risky.

## Failure Modes

The design needs clear behavior for upstream disconnects, duplicate delivery, schema drift, and consumers falling behind. Each is observable, replayable, and isolated from the raw event archive.

## Cost

The primary cost lever is retention: inexpensive object storage keeps the canonical raw stream, while shorter-lived serving tables support fast exploration. Compute scales with sustained event volume and replay frequency.
